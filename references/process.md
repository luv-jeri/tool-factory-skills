# The 9-stage process (full detail)

Read the stage you are on. Each stage has a PASS rule — do not advance until it passes. Checkpoints (in `--checkpoints` mode) are marked ⏸.

## Stage 0 — Reconcile prior research → canonical baseline
**Purpose:** start from existing research, not a blank page; resolve contradictions so the funnel rests on one agreed baseline.
**Actions:** Read all strategy/plan/ranking files with 2 parallel read-only agents (hubs, scoring model, prior recommendations, contradictions). List every contradiction (e.g. finance-first vs dev-first) and resolve via the **canonical-source rule**: README/ADR declares which plan is canonical; stale drafts are superseded IN WRITING. Flag any load-bearing premise that is uncertain or unverifiable from files alone (e.g. "do we already own domain X?") and **ASK THE USER** before relying on it. Mark every prior score ESTIMATED-until-verified; treat the inherited ranking as a hypothesis, never the verdict. **Capture `runway_months`** — how many months before a build must be earning (ADR-0004). It is an explicit per-invocation input, not hardcoded; the opener is judged against it (time-to-traffic vs runway), so confirm it with the user up front.
**Data:** README.md, EXEC-SUMMARY.md, 03-BULLETPROOF-PLAN.md, Money-First plan, 04-IDEA-RESCORE.md, 07-FIRST-TOOL-DECISION.md, 08-TOOL-SELECTION-RUBRIC.md, master ranking spreadsheets, + direct user confirmation on uncertain premises (including `runway_months`).
**PASS:** exactly one canonical hub/plan baseline; all contradictions resolved or escalated; no decision rests on an unverified premise; `runway_months` captured.
**Out:** canonical baseline note (chosen plan, resolved contradictions + superseded drafts, premises confirmed with the user, `runway_months`, candidate hubs to evaluate).

## Stage 1 — Hub selection by pre-flight gate-drill (tool-grade rigor, ADR-0005)
**Purpose:** pick the hub/persona BEFORE sourcing tools, so every candidate shares one audience and an internal-link spine that compounds authority. The hub is the LEAST-reversible decision in the funnel (swap a tool, keep your authority; swap a hub, rebuild from zero), so it earns tool-grade rigor — not a one-shot qualitative judgment.
**Actions:** For each candidate hub, run a **pre-flight gate-drill**: run Gates A–D *quickly* on **2–3 representative candidate tools** in that hub. A hub is blessed only if those drills show **real winnable tools** — evidence, not "plausibly." Enforce a **hard minimum depth ≥ 3 independently-winnable tools** that clear the quick gate-drill; a hub with fewer is rejected outright (depth = the supply of same-hub fallbacks that lets a flopped opener be replaced without abandoning authority). When two hubs are close, **prefer the deeper hub** (more winnable tools = more fallbacks + a longer build runway inside one authority investment). Still prefer low-barrier hubs for a zero-authority brand (low YMYL/E-E-A-T, embed-friendly, B2B-leaning CPC); defer high-RPM YMYL hubs (finance) to a month-12–18 graduation. **Record each hub choice as its own ADR** — it is hard to reverse and a future reader needs the "why this hub" reasoning.
**PASS:** (a) one coherent persona; (b) **≥ 3 tools pass the quick gate-drill on real evidence** (not asserted); (c) low YMYL/E-E-A-T pressure for an anonymous new brand; (d) tools share an internal-link spine. Open with the deepest/most-winnable passing hub; the choice is written up as an ADR.
**Out:** chosen opening hub (+ its ADR) + ranked hub backlog with each hub's gate-drill depth + the link-spine sketch.

## Stage 2 — Standardized candidate sourcing ⏸
**Purpose:** generate ONE bounded, deduped shortlist inside the hub — the fix for the root cause of divergent winners (each agent inherited a different candidate universe).
**Actions:** Pull all tools mapped to the chosen hub from the master ranking; dedupe overlapping ideas to one canonical tool each. Top up to a fixed shortlist size (10–12) with SERP-validated net-new ideas in the same persona. For each candidate record: head term, obvious long-tail variants, intent type (interactive/transactional vs informational), rough buyer slice.
**PASS:** shortlist is a fixed size (target 10–12), fully deduped, every entry in-hub, each carries head term + variants + intent.
**Out:** deduped 10–12 candidate shortlist with per-candidate metadata. **⏸ `--checkpoints`: confirm the shortlist with the user.**

## Stage 3 — Kill-gates A→B→C→D (in order, cheapest first, stop at first cause; DROP or REFUSE)
**Purpose:** eliminate un-buildable/un-rankable/un-monetizable candidates before scoring. The engine (`score.py` `evaluate()`) cites the SINGLE firing cause. Before any gate it **fails closed**: a missing/mistyped required input RAISES, never silently passes (ADR-0009).
- **Gate A — Hard kill (restricted vertical FIRST):** the FIRST check is `adsense_restricted` — an AdSense-restricted vertical (gambling/alcohol/drugs/weapons/adult) is DROPPED no matter how high its CPC (it is unmonetizable, so no axis rescues it — ADR-0009). Then reject if Google answers it natively in-SERP / a browser/OS ships the feature (`native_feature`). *Escape hatch (non-restricted only):* if it can be recast as a multi-step/stateful/file-export tool **that still carries the search demand which justified the candidate**, resurrect it — a recast that keeps the name but loses the searches does not qualify.
- **Gate B — Winnability (decisive, with margin):** pull the live SERP for the head term. A `winnability == 1` is a confident **DROP** only **with margin** — `native_feature`, OR `kd_head ≥ 26` AND `weak_count ≤ 1`, OR an *evidenced* DR-80+ wall (`dr_wall_evidenced`). A `winnability == 1` that sits in the **KD 21–25 noise band** with `weak_count ≥ 2` and no thin-site proof returns **REFUSE** ("pull more SERP/thin-site evidence"), not DROP — a cutoff inside the input's own ±5 KD noise is a coin-flip (ADR-0008). PASS when ≥1 thin/low-DR/one-page site already ranks the head OR long-tail (live newcomer proof) — but to be *honored* the proof must be EVIDENCED (URL + DR + keyword; a bare bool is ignored and flagged, ADR-0009). You buy the long-tail, never the bare head. App-store apps do NOT contest the open-web SERP — count them as absent (app-only "competition" = a weak/empty web SERP = a PASS signal). The DR-wall citation fires only when `dr_wall_evidenced` is set.
- **Gate C — AI-Overview / dead-on-arrival:** `onebox` present → DROP. Then `ai_resistance == 1` → **DROP** ("dead on arrival — AI answers it inline"): informational/single-fact intent answered inline by an AI Overview, or `aio_fire_pct > 85` on a static/info artifact. PASS on transactional "do-it-here" interactive-artifact intent.
- **Cross-field REFUSE (runs before Gate D):** if `cluster_monthly_volume` is below the head bucket's own lower bound (the head implies more searches than the entire cluster), the engine returns **REFUSE** ("re-measure"), not a number — the cluster cannot be below its own head.
- **Gate D — Real-demand floor (MEASURED VOLUME ONLY):** pull ACTUAL keyword volume (Stage 5 method). DROP if `cluster_monthly_volume < 1,000/mo`. **Gate D fires on measured volume ONLY** — NOT on `demand==1`, NOT on the variant count (a thin variant set is a graded demand penalty, not a Gate-D drop), so its citation never mis-blames "volume" when variants were the cause. Run the qualifier-kills-volume test: measure the BROAD commercial term, not just the persona-flavored one (the flavor word can cut volume ~100×). A Gate-D failure is a HARD drop — never rescued by strong Revenue/AI/Build.
- **Finalist bar (NOT a gate):** the **5,000/mo + 100-kw bar is the finalist/commit bar**, not a Gate-D cutoff. A tool between 1,000 and 5,000/mo (or < 100 cluster keywords) passes the gates and **ranks**, but is flagged "below finalist bar" → a fast-follow, not a committable opener (enforced via `first_build_eligible`, ADR-0008).
**PASS:** a candidate must pass A, B, C, D to be scored. A gate cause = DROP (Gate A restricted/native only: try the recast escape hatch for native). An ambiguous/contradictory input = **REFUSE** (a legal first-class outcome, ADR-0001 → go verify, don't pick). Once a cause fires, STOP — do not compute downstream gates or emit any Opportunity number for a dropped/refused candidate. Gates B and D are the most common kills and must cite live evidence.
**Out:** per-candidate gate sheet (A/B/C/D PASS/FAIL/REFUSE + the single firing cause + evidence + who-ranks-page-1) and the surviving contender set.

> **How to run Stage 3 + Stage 4 cheaply:** fan them out with `scripts/research-workflow.js` via the Workflow tool — one researcher + one skeptic per candidate. Run that workflow; do not hand-research 12 tools serially.

## Stage 4 — Weighted scoring on the standard model (verified inputs only) ⏸
**Purpose:** rank survivors objectively on the balanced model, using verified — not guessed — inputs.
**Actions:** Lock the model + weights BEFORE scoring (do not re-tune to fit a favorite): `Opportunity = (0.20·D + 0.25·W + 0.25·AI + 0.20·Rev + 0.10·Build) × 20`. Run `scripts/score.py` on the measured inputs (never hand-score); it computes each dimension on the deterministic anchors in `scoring-model.md` and TAGS each score's evidence tier (real-measured / triangulated / reasoned). Apply the veto: any single 1 disqualifies a first build. **Evidence-tier-gated committability (ADR-0009): RANK on triangulated data, but COMMIT only on real-measured.** A tool ranks at Stage 4 on triangulated/SERP-density inputs, but `first_build_eligible` is true only when Demand + Winnability + AI-Resistance are all `real-measured` (and it is at/above the finalist bar, within runway, and carries no `sensitivity:` flag). Below that it is rankable-but-not-committable until Stage 5 upgrades the evidence — this is the rank-on-estimates / commit-on-real-data funnel, not a contradiction.
**PASS:** ranked by Opportunity (OK > VETO > REFUSE > DROP, then score, then name), BUT the first build must score 4–5 on EVERY dimension (no fatal weakness) AND be `first_build_eligible`. Pick the all-rounder, not the highest-on-one-axis trap.
**Out:** sortable scorecard (0–100 + per-dimension 1–5 + evidence tier + `first_build_eligible`), provisional winner + ranked runner-ups. **⏸ `--checkpoints`: confirm the finalists before the real-data pull.**

## Stage 5 — Real-demand verification (empirical, BLOCKING gate)
**Purpose:** replace every estimated/triangulated volume on the finalists with real clickstream data — the decisive gate that has historically flipped picks.
**Actions:** Use the fixed free toolchain (`free-tools.md`): Ahrefs free Keyword Generator at `ahrefs.com/keyword-generator/?country=us` (or a paid Ahrefs/Semrush seat if available), driven in-browser. For each finalist pull banded Volume, KD (Easy/Med/Hard), and total cluster size ("X of N keywords"). **Throttle workaround:** the results component caches after ~2 lookups — RELOAD the page fresh before each new keyword (re-submitting in place returns stale rows). Re-run the qualifier test on real data (broad vs persona term); read KD as a hub-wide reality check (money heads usually "Hard" → head ≈ year-2, long-tail ≈ months 3–12).
**Fail closed (ADR-0009):** **no real data ⇒ REFUSE, never estimate.** A broken/empty scrape is NOT low volume — distinguish the two and **hard-fail loudly** rather than reading an empty result as "0/mo" and dropping a good tool (or passing a bad one). Triangulate **≥ 2 free sources** before trusting a number; for the actual COMMIT, recommend a **paid seat** (the engine never downgrades an evidence tier to clear the commit bar — a tool stays rankable-but-not-committable until real data lands).
**PASS:** a finalist survives only if real cluster volume ≥ ~5,000/mo on the broad commercial term with ≥ ~100 long-tail keywords. If real data contradicts the scored Demand, re-score and re-rank (this is where freelance-rate <100/mo dies and invoice/timecard >10k/mo wins).
**Out:** real keyword-volume table per finalist (band, KD, cluster size, source/date); the re-confirmed or re-ordered winner.

## Stage 6 — Adversarial "kill the winner" ⏸
**Purpose:** red-team the chosen tool's load-bearing claims to break it or harden it before committing build effort.
**Actions:** Run a SEPARATE skeptic (a distinct agent) that independently re-checks the 2–3 most decision-critical, most-hallucination-prone claims: (1) real demand/volume, (2) AI-Overview presence on the head term, (3) the revenue-to-buyer-slice assumption. Do NOT abbreviate this into a self-review — the winner earns a full separate-skeptic pass even when it "looks obviously fine." For Revenue, force "who in this audience actually purchases?" and DISCOUNT affiliate to that slice. Adjust scores DOWN where evidence is weaker; record every honest correction. Re-check for a false tiebreaker / stale premise (how the bogus "you own contractrates.fyi" tiebreaker was caught) — cross-check load-bearing premises against the canonical plan and the user; supersede the bad draft in writing.
- **Future-SERP conservative haircut (ADR-0006):** the three decaying dimensions — AI-Resistance, Winnability, Demand — pay off only when you rank 6–12 months out, so score them against the **rank-time SERP, not today's snapshot.** Project `aio_fire_pct` up (AIOs creep down-funnel into commercial terms); if page 1 shows fresh clones launched in the last few months, treat the niche as contesting and haircut Winnability (a visibly eroding moat is not a moat); require a non-negative demand trend, not just an absolute volume. Scores should read *lower* than raw data suggests — by design (this is ADR-0001 applied to time, not pessimism). Interactive/personalized artifacts keep AI-Resistance 5 because Google structurally can't run the tool inline — bias the opener toward them.
- **Time-to-rank vs runway (ADR-0004):** estimate time-to-first-traffic from head KD + thin-site-proof long-tail speed + domain sandbox age, and compare to the `runway_months` captured at Stage 0. If `est_time_to_traffic_months > runway_months`, the tool is a fast-follow, not the opener (a winnable-but-slow tool that matures past the runway is the fatal outcome itself). The opener is biased toward fastest-defensible-time-to-traffic, NOT peak Opportunity — a higher-ceiling-but-slower tool becomes build #2 once revenue de-risks the wait.
**PASS:** the winner STANDS only if all load-bearing claims survive (after at most documented corrections) AND plausibly ranks inside the runway. Any claim that fails outright promotes the top same-hub runner-up and re-runs Stages 5–6 on it.
**Out:** red-team verdict (each claim survived / corrected / failed) + the honest corrections folded into final scores and monetization stance (e.g. display-first). **⏸ `--checkpoints`: confirm the winner before deliverables.**

## Stage 7 — Confidence, fallback & monitoring
**Purpose:** make the recommendation honest and robust.
**Actions:** Tag each load-bearing claim with a confidence tier (real-measured=high; triangulated=medium; reasoned=low) + dates/sources. Name an explicit SAME-HUB fallback (top runner-up) + the SWITCH TRIGGER that would promote it (real volume < ~5k/mo cluster; the red-team breaks a claim; a DR-80 entrant captures the long-tail). The Stage-7 same-hub fallback is reframed as **roadmap position #2** (ADR-0007). Add a monitor-list with cadence (AI-resistance "true today, monitor"; demand re-verify in a paid seat; ad-network thresholds). List the gating FIRST build action that closes any remaining low-confidence gap. **Distinguish the opener from the highest-ceiling tool:** when they differ, name both — the opener is chosen for fastest-defensible-time-to-traffic (survival), the higher-ceiling tool is a later build once revenue de-risks the wait (ADR-0004).
**PASS:** the winner has a stated confidence per load-bearing claim, a named same-hub fallback with a switch trigger, and a monitor-list. No bare single-bet recommendations.

## Stage 8 — Deliverables
**Purpose:** emit reusable, auditable artifacts so the decision is reproducible and the build can start. See `deliverables.md` for templates.
**Actions:** Write the decision brief (hub + tool + why, with the divergence-from-internal-ranking explained and the honest corrections). **The deliverable is the next build PLUS a short 3–4-step ordered roadmap (ADR-0007)** — the opener (fastest-defensible-time-to-traffic) foregrounded, then follow-ons ordered to compound authority on the link-spine and climb the opportunity ceiling once revenue de-risks the wait. The roadmap is a re-derivable sketch, not a frozen plan; the single *rigorous* decision each run remains "what to build NOW." **State both the opener and the highest-ceiling tool when they differ** (ADR-0004). Ship the live-formula scored spreadsheet. Attach the real keyword-volume table and the red-team verdict. Write the starter build brief (differentiators that ARE the AI-resistance moat, one-variant-per-URL long-tail page roadmap, monetization slots, correctness/unit-test list). Log the decision + scoring model to project memory; supersede any stale draft in writing.
**PASS:** all artifacts exist (brief, scorecard, real-volume table, red-team verdict, build brief, 3–4-step ordered roadmap) AND the decision is logged to memory with superseded drafts noted.

## The hub-gate criteria (Stage 1, ADR-0005)
A hub PASSES only if ALL hold: (a) one coherent persona; (b) **≥ 3 tools clear the pre-flight gate-drill on real evidence** — run Gates A–D *quickly* on 2–3 representative tools; depth ≥ 3 is a hard minimum (it is the supply of same-hub fallbacks), and "plausibly clears the gates" no longer qualifies — it must be drilled; (c) low YMYL/E-E-A-T pressure for an anonymous new brand; (d) tools share an internal-link spine. Prefer the **deeper** hub when two are close. Open with the most winnable passing hub and **record the choice as its own ADR**; graduate to higher-RPM YMYL hubs (finance) at month 12–18.

## Contradiction-reconciliation rule (Stage 0)
When sources conflict: (1) the README/ADR names which plan is canonical — it wins; (2) a load-bearing premise that can't be settled from files (domain ownership, budget, prior commitments) is escalated to the user, never assumed; (3) the stale/losing draft is superseded IN WRITING (don't leave two conflicting files). This is exactly how the false "you own contractrates.fyi" tiebreaker and the finance-first-vs-dev-first conflict were caught.
