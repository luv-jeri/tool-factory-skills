# The deliverable is the next build plus a short ordered roadmap, not a lone pick

**Status:** accepted

Tools in a hub are not independent — they share a link-spine and cross-link, so build *order* compounds topical authority, and ADR-0004 already split the answer into a survival-first opener vs a higher-ceiling later build. A single-winner output hides that sequence and reproduces the original "overlapping/duplicate ideas" confusion, where overlapping tools were mistaken for rivals for one slot when they were really sequence positions in one hub.

## Decision

The funnel outputs **the next build, decided with full rigor, *plus* its position in a short 3–4-step roadmap**:
- **Opener** — chosen for fastest-defensible-time-to-traffic / survival (ADR-0004).
- **Follow-ons** — ordered to compound authority on the link-spine, climb the opportunity ceiling once revenue de-risks the wait, and preserve fallback depth (ADR-0005).

The roadmap is a re-derivable sketch, never a frozen plan (data changes every run). The single *rigorous* decision each invocation remains "what to build **now**"; the Stage-7 "same-hub fallback" is reframed as roadmap position #2.

## Consequences

- Deliverables (`deliverables.md`) gain a short roadmap artifact; the decision brief states both the opener and the highest-ceiling tool when they differ (ADR-0004).
- "Pick the next tool" requests are answered with the now-pick foregrounded and the sequence as supporting context — not scope creep, because order is a survival decision under runway pressure.
